SOLID 五大原则 —— Spring Boot 实战详解

SOLID 是面向对象设计中五个核心原则的缩写，由 Robert C. Martin（Uncle Bob）提出。
它们不是语法规则，而是指导你如何拆分职责、设计接口、组织依赖的架构准则。
在 Spring Boot 项目中，SOLID 直接决定了：代码是否易测试、是否易扩展、是否会在需求变更时「牵一发而动全身」。

目录​

  1. S — 单一职责原则（SRP）
  2. O — 开闭原则（OCP）
  3. L — 里氏替换原则（LSP）
  4. I — 接口隔离原则（ISP）
  5. D — 依赖倒置原则（DIP）
  6. SOLID 在 Spring Boot 分层中的整体映射
  7. 常见反模式速查表

S — 单一职责原则（SRP）​

定义​

一个类应该只有一个引起它变化的原因。

换句话说：一个类/模块只负责一类业务职责。当需求变更时，修改应该局限在少数几个类中，而不是一个「上帝类」里到处改。

违反示例：上帝 Service​

@Service
public class UserService {

    public User register(RegisterDTO dto) { /* 注册逻辑 */ }

    public void sendWelcomeEmail(User user) { /* 发邮件 */ }

    public String generateJwt(User user) { /* 生成 Token */ }

    public void exportUsersToExcel(HttpServletResponse response) { /* 导出 Excel */ }

    public void syncUserToElasticsearch(User user) { /* 同步 ES */ }
}

问题：

  • 注册规则变了 → 改 UserService
  • 邮件模板变了 → 改 UserService
  • JWT 算法升级了 → 改 UserService
  • 导出格式从 Excel 改成 CSV → 还是改 UserService

一个类承担了：用户领域逻辑、通知、认证、报表、搜索同步 五种职责，违反 SRP。

符合 SRP 的 Spring Boot 拆分​

// 只负责用户领域逻辑
@Service
@RequiredArgsConstructor
public class UserService {
    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;

    public User register(RegisterDTO dto) {
        User user = new User();
        user.setUsername(dto.getUsername());
        user.setPassword(passwordEncoder.encode(dto.getPassword()));
        return userRepository.save(user);
    }
}

// 只负责通知
@Service
@RequiredArgsConstructor
public class EmailNotificationService {
    private final JavaMailSender mailSender;

    public void sendWelcomeEmail(User user) {
        // 构造并发送邮件
    }
}

// 只负责 Token
@Service
public class JwtTokenService {
    public String generateToken(User user) { /* ... */ }
}

// 只负责导出
@Service
public class UserExportService {
    public void exportToExcel(List<User> users, OutputStream out) { /* ... */ }
}

Controller 编排多个单一职责的 Service：

@RestController
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;
    private final EmailNotificationService emailService;
    private final JwtTokenService jwtTokenService;

    @PostMapping("/register")
    public RegisterVO register(@RequestBody RegisterDTO dto) {
        User user = userService.register(dto);
        emailService.sendWelcomeEmail(user);
        String token = jwtTokenService.generateToken(user);
        return new RegisterVO(user.getId(), token);
    }
}

Spring Boot 中的典型应用场景​

层级单一职责体现
Controller只做参数校验、调用 Service、封装响应，不写业务逻辑
Service一个 Service 对应一个聚合根或一个业务用例（如 OrderPayService）
Repository / Mapper只做持久化，不夹杂业务判断
DTO / VO / EntityEntity 映射表结构；DTO 承载入参；VO 承载出参，不混用
配置类RedisConfig 只配 Redis；SecurityConfig 只配安全，不塞业务 Bean

与项目结构的结合（以 super-system 为例）​

当前项目中已有较好的拆分雏形：

system/
├── SysConfig.java           # 实体：只描述数据
├── SysConfigMapper.java     # 持久化：只负责数据库访问
├── SysConfigServiceImpl.java # 领域服务：只负责 sys_config 业务
└── SysConfigController.java  # 接口层：只负责 HTTP 交互

若把「参数校验 + 缓存刷新 + 操作日志」都写进 SysConfigServiceImpl，就违反了 SRP。正确做法是：

  • SysConfigValidator — 校验 configKey 格式
  • SysConfigCacheService — 刷新 Redis 缓存
  • SysConfigAuditListener — 记录变更日志（可用 @EventListener）

判断口诀​

问自己：「这个类因为哪几种不同的需求变更而需要修改？」 如果答案超过一种，就该拆分。

O — 开闭原则（OCP）​

定义​

软件实体应对扩展开放，对修改关闭。

新增功能时，尽量通过新增类/实现来完成，而不是修改已有稳定代码。

违反示例：支付方式 if-else 地狱​

@Service
public class PaymentService {

    public void pay(String payType, BigDecimal amount) {
        if ("ALIPAY".equals(payType)) {
            // 支付宝逻辑
        } else if ("WECHAT".equals(payType)) {
            // 微信逻辑
        } else if ("UNIONPAY".equals(payType)) {
            // 银联逻辑 —— 每新增一种支付方式都要改这个方法
        }
    }
}

每接入一种新支付渠道，都要改 PaymentService，违反 OCP，也容易引入回归 Bug。

符合 OCP：策略模式 + Spring 自动注入​

1. 定义抽象策略

public interface PaymentStrategy {
    /** 是否支持该支付类型 */
    boolean supports(String payType);
    /** 执行支付 */
    PayResult pay(PayRequest request);
}

2. 各渠道独立实现（扩展时新增类，不改旧代码）

@Component
public class AlipayStrategy implements PaymentStrategy {
    @Override
    public boolean supports(String payType) {
        return "ALIPAY".equals(payType);
    }
    @Override
    public PayResult pay(PayRequest request) { /* 支付宝 SDK 调用 */ }
}

@Component
public class WechatPayStrategy implements PaymentStrategy {
    @Override
    public boolean supports(String payType) {
        return "WECHAT".equals(payType);
    }
    @Override
    public PayResult pay(PayRequest request) { /* 微信支付 */ }
}

3. 上下文通过构造器注入所有策略

@Service
@RequiredArgsConstructor
public class PaymentService {

    private final List<PaymentStrategy> strategies;

    public PayResult pay(String payType, PayRequest request) {
        return strategies.stream()
                .filter(s -> s.supports(payType))
                .findFirst()
                .orElseThrow(() -> new BusinessException("不支持的支付方式: " + payType))
                .pay(request);
    }
}

新增「数字人民币」支付时，只需新增 DigitalRmbStrategy implements PaymentStrategy，无需修改 PaymentService。

Spring Boot 中 OCP 的常见实现手段​

手段示例
策略模式支付、物流、消息推送渠道
模板方法AbstractJobHandler 定义执行骨架，子类实现具体步骤
Spring 事件订单创建后发布 OrderCreatedEvent，各监听器独立扩展
@ConditionalOnProperty同一接口多种实现，按配置切换，不改调用方
SPI / spring.factoriesStarter 扩展点，框架对扩展开放
AOP 切面日志、权限、限流横切逻辑，不侵入业务方法

实战：消息通知扩展​

// 事件定义
public record UserRegisteredEvent(Long userId, String email) {}

// 发布方 — 稳定不变
@Service
@RequiredArgsConstructor
public class UserService {
    private final ApplicationEventPublisher eventPublisher;

    public User register(RegisterDTO dto) {
        User user = /* ... */;
        eventPublisher.publishEvent(new UserRegisteredEvent(user.getId(), user.getEmail()));
        return user;
    }
}

// 扩展 1：发邮件（已有）
@Component
public class WelcomeEmailListener {
    @EventListener
    public void onUserRegistered(UserRegisteredEvent event) { /* 发邮件 */ }
}

// 扩展 2：新增短信通知 —— 只加监听器，不改 UserService
@Component
public class WelcomeSmsListener {
    @EventListener
    public void onUserRegistered(UserRegisteredEvent event) { /* 发短信 */ }
}

注意边界​

OCP 不是「永远不能改旧代码」。当抽象设计本身有误时，修改基类是合理的。OCP 的目标是：在可预见的扩展方向上预留抽象，减少高频变更点的修改次数。

L — 里氏替换原则（LSP）​

定义​

子类对象必须能够替换其父类对象，且程序行为不变。

子类可以扩展父类能力，但不能破坏父类与调用方之间的契约（前置条件不能加强、后置条件不能削弱、不变量必须保持）。

违反示例：正方形继承长方形​

经典教科书例子，映射到业务：

public class DiscountPolicy {
    public BigDecimal apply(BigDecimal originalPrice) {
        // 约定：originalPrice 必须 > 0，返回价格 >= 0
        return originalPrice.multiply(new BigDecimal("0.9"));
    }
}

public class FullReductionPolicy extends DiscountPolicy {
    @Override
    public BigDecimal apply(BigDecimal originalPrice) {
        // 破坏了父类契约：原价 < 100 时直接抛异常
        if (originalPrice.compareTo(new BigDecimal("100")) < 0) {
            throw new IllegalArgumentException("满减活动最低消费 100 元");
        }
        return originalPrice.subtract(new BigDecimal("20"));
    }
}

调用方按 DiscountPolicy 契约使用：

public BigDecimal calculate(List<DiscountPolicy> policies, BigDecimal price) {
    BigDecimal result = price;
    for (DiscountPolicy policy : policies) {
        result = policy.apply(result); // 期望任何 DiscountPolicy 都能处理任意正数价格
    }
    return result;
}

当 FullReductionPolicy 替换进来时，小额订单直接抛异常 —— 无法安全替换父类，违反 LSP。

符合 LSP 的重构​

方案一：子类遵守父类契约，额外约束通过文档/校验层处理

public class FullReductionPolicy extends DiscountPolicy {
    @Override
    public BigDecimal apply(BigDecimal originalPrice) {
        if (originalPrice.compareTo(new BigDecimal("100")) < 0) {
            return originalPrice; // 不满足条件时原样返回，不抛异常
        }
        return originalPrice.subtract(new BigDecimal("20"));
    }
}

方案二（更推荐）：不用继承，用组合 + 接口

public interface DiscountPolicy {
    boolean isApplicable(BigDecimal price);
    BigDecimal apply(BigDecimal price);
}

@Component
public class FullReductionPolicy implements DiscountPolicy {
    @Override
    public boolean isApplicable(BigDecimal price) {
        return price.compareTo(new BigDecimal("100")) >= 0;
    }
    @Override
    public BigDecimal apply(BigDecimal price) {
        return price.subtract(new BigDecimal("20"));
    }
}

Spring Boot 中的典型 LSP 场景​

1. UserDetailsService 实现​

public interface UserDetailsService {
    UserDetails loadUserByUsername(String username) throws UsernameNotFoundException;
}

自定义实现必须保证：

  • 用户名存在 → 返回完整的 UserDetails
  • 用户名不存在 → 抛 UsernameNotFoundException（不能返回 null，不能抛其他未声明异常）
@Service
@RequiredArgsConstructor
public class DbUserDetailsService implements UserDetailsService {
    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String username) {
        User user = userRepository.findByUsername(username)
                .orElseThrow(() -> new UsernameNotFoundException("用户不存在: " + username));
        return User.builder()
                .username(user.getUsername())
                .password(user.getPassword())
                .roles(user.getRole())
                .build();
    }
}

若子类实现返回 null 或吞掉异常，Spring Security 过滤器链行为异常，即违反 LSP。

2. MyBatis-Plus ServiceImpl 继承​

@Service
public class SysConfigServiceImpl extends ServiceImpl<SysConfigMapper, SysConfig> {
    // 可以新增业务方法，但不要重写 save/remove 并改变其语义
    // 例如：重写 save 使其静默丢弃 null 值 —— 会破坏调用方对 ServiceImpl 的期望
}

继承 ServiceImpl 时，调用方期望 save 会持久化实体。若子类重写后改为「只写缓存不写库」，则无法替换父类使用。

3. 自定义 HttpServletRequestWrapper​

public class CachedBodyRequestWrapper extends HttpServletRequestWrapper {
    // 必须保证 getInputStream() 可重复读，且与父类其他方法行为一致
}

过滤器中若用子类替换原始 Request，后续 Filter / Controller 必须能无差别使用。

LSP 检查清单​

  • 子类是否抛出了父类未声明的受检异常？
  • 子类是否加强了入参约束（父类接受 null，子类不接受）？
  • 子类是否削弱了返回值（父类承诺非 null，子类可能返回 null）？
  • 子类是否改变了方法副作用（父类只读，子类偷偷写库）？

I — 接口隔离原则（ISP）​

定义​

客户端不应被迫依赖它不使用的方法。

优先使用小而专一的接口，而不是臃肿的「万能接口」。在 Java 中，表现为：不强迫实现类实现一堆空方法或 throw new UnsupportedOperationException()。

违反示例：臃肿的「万能」用户接口​

public interface UserService {
    User getById(Long id);
    List<User> listAll();
    void create(User user);
    void update(User user);
    void delete(Long id);
    void exportExcel();
    void importExcel(InputStream in);
    void resetPassword(Long id);
    void bindWechat(Long id, String openId);
    void syncToDingTalk(Long id);
}

一个只需要「查询用户」的报表模块，却被迫依赖整个 UserService，甚至间接依赖微信、钉钉相关逻辑。

符合 ISP：按客户端需求拆分接口​

// 读操作 — 报表、其他服务只需要这个
public interface UserQueryService {
    User getById(Long id);
    List<User> listAll();
}

// 写操作 — 管理后台
public interface UserManageService {
    void create(User user);
    void update(User user);
    void delete(Long id);
}

// 账号安全 — 认证模块
public interface UserAccountService {
    void resetPassword(Long id);
}

// 第三方绑定 — 集成模块
public interface UserSocialBindService {
    void bindWechat(Long id, String openId);
}

实现类可以合并实现多个小接口：

@Service
@RequiredArgsConstructor
public class UserServiceImpl implements UserQueryService, UserManageService, UserAccountService {
    private final UserRepository userRepository;
    // ...
}

报表模块只注入 UserQueryService：

@Service
@RequiredArgsConstructor
public class ReportService {
    private final UserQueryService userQueryService; // 不依赖写操作、不依赖微信绑定
}

Spring Boot 中的典型 ISP 场景​

1. Repository 继承 JpaRepository 的反思​

JpaRepository 有 20+ 方法。若某个领域只需要自定义查询：

// 不好：暴露了一堆用不到的 CRUD
public interface OrderRepository extends JpaRepository<Order, Long> { }

// 更好：只继承需要的能力
public interface OrderRepository extends Repository<Order, Long> {
    Optional<Order> findById(Long id);
    Order save(Order order);
    @Query("select o from Order o where o.status = :status")
    List<Order> findByStatus(@Param("status") String status);
}

2. Feign Client 按调用方拆分​

// 订单服务调用用户服务 —— 只需要基础信息
@FeignClient(name = "user-service", contextId = "userBasicClient")
public interface UserBasicClient {
    @GetMapping("/users/{id}")
    UserBasicVO getBasicInfo(@PathVariable Long id);
}

// 消息服务调用用户服务 —— 只需要联系方式
@FeignClient(name = "user-service", contextId = "userContactClient")
public interface UserContactClient {
    @GetMapping("/users/{id}/contact")
    UserContactVO getContact(@PathVariable Long id);
}

避免一个 Feign 接口包含用户服务的全部 API，导致调用方依赖无关 DTO 和降级逻辑。

3. Spring Security 的接口拆分​

Spring Security 本身贯彻 ISP：

  • AuthenticationManager — 认证
  • AccessDecisionManager — 授权决策
  • UserDetailsService — 加载用户信息
  • PasswordEncoder — 密码编码

Security 配置类按需注入，不必实现一个包含所有安全方法的巨型接口。

4. 回调接口 vs 函数式接口​

// 臃肿
public interface FileProcessor {
    void onStart();
    void onProgress(int percent);
    void onComplete();
    void onError(Exception e);
}

// 调用方若只关心完成，却被迫实现全部方法

使用 Java 函数式接口或 Spring 的 Consumer / 可选回调：

public void processFile(MultipartFile file, Consumer<FileResult> onComplete) {
    // ...
}

ISP 与 SRP 的关系​

  • SRP：一个类只有一个职责
  • ISP：一个接口只服务于一类客户端

两者协同：拆分接口的同时，实现类也更聚焦。

D — 依赖倒置原则（DIP)​

定义​

高层模块不应依赖低层模块，二者都应依赖抽象。
抽象不应依赖细节，细节应依赖抽象。

「高层」= 业务逻辑（Service、Domain）；「低层」= 基础设施（数据库、MQ、第三方 API）。DIP 要求业务层面向接口编程，具体实现由 Spring 容器在运行时注入。

违反示例：Service 直接依赖具体实现​

@Service
public class OrderService {

    public void createOrder(OrderDTO dto) {
        // 直接 new 具体实现，强耦合 MySQL
        OrderRepository repository = new JdbcOrderRepository();
        repository.save(dto);

        // 直接调用阿里云 SMS SDK
        AliyunSmsClient sms = new AliyunSmsClient("ak", "sk");
        sms.send(dto.getPhone(), "下单成功");
    }
}

换 PostgreSQL、换腾讯云短信，都要改 OrderService —— 高层依赖了低层细节。

符合 DIP：依赖抽象 + Spring 注入​

// 抽象 — 定义在领域层 / api 模块
public interface OrderRepository {
    void save(Order order);
}

public interface SmsSender {
    void send(String phone, String message);
}

// 高层只依赖接口
@Service
@RequiredArgsConstructor
public class OrderService {
    private final OrderRepository orderRepository;
    private final SmsSender smsSender;

    public void createOrder(OrderDTO dto) {
        Order order = Order.from(dto);
        orderRepository.save(order);
        smsSender.send(dto.getPhone(), "下单成功");
    }
}

// 低层实现细节 — 可替换
@Repository
public class MybatisOrderRepository implements OrderRepository {
    private final OrderMapper orderMapper;
    @Override
    public void save(Order order) { orderMapper.insert(order); }
}

@Component
@ConditionalOnProperty(name = "sms.provider", havingValue = "aliyun")
public class AliyunSmsSender implements SmsSender { /* ... */ }

@Component
@ConditionalOnProperty(name = "sms.provider", havingValue = "tencent")
public class TencentSmsSender implements SmsSender { /* ... */ }

切换短信供应商只需改配置 sms.provider=tencent，OrderService 零修改。

Spring Boot 如何天然支持 DIP​

Spring 的 IoC（控制反转） 就是 DIP 的工程实现：

Spring 机制DIP 体现
@Autowired / 构造器注入依赖接口，运行时注入实现
@Configuration + @Bean创建具体 Bean，对业务层透明
spring-boot-starter-*自动配置隐藏底层细节（Tomcat、HikariCP）
@ConditionalOnClass按 classpath 选择实现，业务代码无感
Spring Cloud OpenFeign远程调用抽象为接口，屏蔽 HTTP 细节

实战：存储服务可替换设计​

// api 模块 — 抽象
public interface FileStorageService {
    String upload(String filename, InputStream content);
    InputStream download(String fileKey);
}

// 实现 1：本地磁盘
@Service
@ConditionalOnProperty(name = "storage.type", havingValue = "local", matchIfMissing = true)
public class LocalFileStorageService implements FileStorageService { /* ... */ }

// 实现 2：MinIO
@Service
@ConditionalOnProperty(name = "storage.type", havingValue = "minio")
public class MinioFileStorageService implements FileStorageService { /* ... */ }

// 实现 3：阿里云 OSS
@Service
@ConditionalOnProperty(name = "storage.type", havingValue = "oss")
public class OssFileStorageService implements FileStorageService { /* ... */ }

业务 Service：

@Service
@RequiredArgsConstructor
public class AvatarService {
    private final FileStorageService fileStorageService; // 只认接口

    public String uploadAvatar(MultipartFile file) {
        return fileStorageService.upload(file.getOriginalFilename(), file.getInputStream());
    }
}

结合本项目的 DIP 实践建议​

当前 SysConfigController 直接依赖具体类 SysConfigServiceImpl：

@RequiredArgsConstructor
public class SysConfigController {
    private final SysConfigServiceImpl sysConfigService; // 依赖具体实现
}

更符合 DIP 的写法：

// 定义业务接口
public interface SysConfigService extends IService<SysConfig> {
    List<SysConfig> listVisibleConfigs();
}

@Service
public class SysConfigServiceImpl extends ServiceImpl<SysConfigMapper, SysConfig>
        implements SysConfigService {
    @Override
    public List<SysConfig> listVisibleConfigs() { /* ... */ }
}

@RestController
@RequiredArgsConstructor
public class SysConfigController {
    private final SysConfigService sysConfigService; // 依赖抽象
}

好处：

  • 单元测试可 Mock SysConfigService
  • 未来若有缓存装饰实现 CachingSysConfigService implements SysConfigService，Controller 无需修改

DIP vs 简单注入​

DIP 不是「凡是用 @Autowired 就算 DIP」。 如果注入的仍是具体类（AliyunSmsClient），只是换了创建方式，耦合依旧。

判断标准：高层模块的字段类型是接口还是具体类？

SOLID 在 Spring Boot 分层中的整体映射​

┌─────────────────────────────────────────────────────────────┐
│  Controller 层                                               │
│  职责：HTTP 适配（SRP）                                       │
│  依赖：Service 接口（DIP）                                    │
├─────────────────────────────────────────────────────────────┤
│  Service 层                                                  │
│  职责：业务用例编排（SRP）                                    │
│  扩展：策略 / 事件 / 模板方法（OCP）                          │
│  依赖：Repository / 外部服务接口（DIP）                       │
├─────────────────────────────────────────────────────────────┤
│  Repository / Mapper 层                                    │
│  职责：持久化（SRP）                                          │
│  接口：只暴露必要查询方法（ISP）                              │
├─────────────────────────────────────────────────────────────┤
│  Infrastructure 层                                         │
│  实现：DB / Redis / MQ / 第三方 SDK（DIP 的细节层）           │
│  替换：@ConditionalOnProperty / 多实现（LSP + OCP）         │
└─────────────────────────────────────────────────────────────┘

一个完整业务流示例：创建系统配置​

// ISP + DIP：小而专一的接口
public interface SysConfigRepository {
    void save(SysConfig config);
    Optional<SysConfig> findByKey(String key);
}

public interface CacheEvictor {
    void evict(String cacheKey);
}

// SRP：只负责创建配置业务
@Service
@RequiredArgsConstructor
public class SysConfigCreateService {
    private final SysConfigRepository repository;
    private final CacheEvictor cacheEvictor;
    private final ApplicationEventPublisher eventPublisher;

    public void create(CreateConfigDTO dto) {
        SysConfig config = SysConfig.from(dto);
        repository.save(config);
        cacheEvictor.evict("sys:config:" + config.getConfigKey());
        eventPublisher.publishEvent(new ConfigCreatedEvent(config));
    }
}

// OCP：新增审计 —— 加监听器，不改 CreateService
@Component
public class ConfigAuditListener {
    @EventListener
    public void onConfigCreated(ConfigCreatedEvent event) { /* 写审计日志 */ }
}

常见反模式速查表​

反模式违反原则典型表现重构方向
上帝类SRP一个 Service 上千行，什么都有按用例/领域拆分
if-else 类型判断OCP每加一种类型改一处策略模式 + Spring 注入 List
继承滥用LSP子类改变父类语义组合优于继承；接口 + 多实现
胖接口ISP实现类大量空方法拆分为多个小接口
直接 new 实现类DIPnew XxxClient() 写在 Service 里提取接口 + @Bean / @Component
Controller 写业务SRPController 里 SQL、事务、发邮件下推到 Service
Entity 当 DTO 用SRPController 直接返回 Entity使用 VO/DTO 隔离
具体类注入DIPprivate final XxxServiceImpl改为接口类型
工具类静态方法泛滥DIP/OCPXxxUtils.doEverything()有状态/可替换逻辑做成 Bean

总结​

原则一句话Spring Boot 实践要点
S 单一职责一个类只管一件事Controller/Service/Repository 各司其职
O 开闭加功能少改旧代码策略模式、Spring Event、@Conditional
L 里氏替换子类能无缝替换父类实现 Spring 扩展接口时遵守契约
I 接口隔离接口要小而专Feign/Repository/Service 按调用方拆分
D 依赖倒置面向接口，Spring 注入构造器注入接口；Starter 隐藏实现细节

SOLID 五条原则彼此关联：

  • 做好 SRP，才容易拆分出合理的接口（ISP）
  • 有了抽象接口，才能实现 DIP 和 OCP
  • OCP 的多态扩展要建立在 LSP 保证的契约之上

在 Spring Boot 项目中，不必教条地「一个类一个方法」，而应在需求变更的频率和方向上做出预判，让代码在扩展时付出的代价最小。这就是 SOLID 在工程中的真正价值。
